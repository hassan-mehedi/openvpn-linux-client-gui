# Feature Parity Matrix

| Capability | Parity label | Initial status | Notes |
| --- | --- | --- | --- |
| Profile import from file | `parity-direct` | initial UI and CLI implemented | Core onboarding service, profile catalog, GTK file import, and CLI path exist. |
| Profile import from URL | `parity-direct` | initial UI and CLI implemented | HTTPS validation, duplicate detection, GTK URL import, and CLI path exist. |
| Token URL onboarding | `parity-linux-adapted` | initial UI and CLI implemented | `openvpn://import-profile/...` is normalized into an HTTPS import flow in core, GTK, and CLI paths. |
| Connection lifecycle state machine | `parity-direct` | initial orchestration implemented | Core transitions, validation, and session lifecycle coordination exist. |
| Session telemetry and stats | `parity-linux-adapted` | initial implementation | The profiles view now shows best-effort bytes, rates, packet counters, latency, and a rolling throughput graph from typed session telemetry snapshots. |
| App settings model | `parity-linux-adapted` | initial GUI implemented | Strong typing and XDG config persistence exist, the GTK shell exposes the full settings surface with DCO capability-aware gating, and the connect path now applies a broader validated subset: backend overrides for `proto-override`, `dns-fallback-google`, `persist-tun`, TLS 1.3 enforcement (`tls-version-min=1.3`), strict `security_level` hardening (`enable-legacy-algorithms=false`, `allow-compression=no`), Linux-adapted `block_ipv6` (`ipv6=no`), Linux-adapted `local_dns` scope handling (`dns-scope=global` when disabled), and `dco`, plus app-side `connection_timeout` budgeting across active connect/reconnect phases. Backend-dependent features like `dns-scope` still surface backend errors when the host resolver integration does not support them. |
| Secure saved passwords | `parity-linux-adapted` | initial implementation | Profile details now surface saved-password state, the attention dialog can store passwords securely, and the session lifecycle auto-consumes saved passwords when a single password prompt is detected. |
| Profile management | `parity-linux-adapted` | initial GUI implemented | The profile details dialog now supports local name/proxy override reset, shows import origin and backend metadata such as validation/read-only state, usage count, tags, and timestamps, and keeps these edits in the core catalog layer instead of the UI. |
| Proxy management | `parity-direct` | initial GUI implemented | Saved proxy CRUD, per-profile assignment, CLI assignment/remove paths, and connect-time proxy override application now exist. Authenticated proxies fail explicitly when assigned credentials are unavailable. |
| Diagnostics export | `parity-linux-adapted` | initial GUI implemented | Report building and redaction logic exist, and the GTK shell now exposes a diagnostics center with reachability, capability, environment checks, posture-aware troubleshooting guidance, live redacted session-log streaming, and support bundle export. Deeper guided workflows are still pending. |
| GTK profile list UI | `parity-direct` | initial implementation | Main window lists profiles, supports search, import, refresh, connect, and delete actions. |
| Connect flow UI | `parity-direct` | challenge flow implemented | Main window now follows active session state, surfaces explicit connect/pause/resume/disconnect and recovery actions, keeps connecting/reconnecting/paused/error states visible in the summary shell, drives multi-field credential/challenge prompts through the verified OpenVPN 3 input queue, and can save password-style prompts through the secure secret-store boundary. |
| GUI navigation shell | `parity-linux-adapted` | initial implementation | The desktop shell now has dedicated Profiles, Settings, and Diagnostics views instead of a single-screen layout. |
| Native packaging | `parity-linux-adapted` | deferred | Packaging directories are prepared for DEB/RPM work. |
